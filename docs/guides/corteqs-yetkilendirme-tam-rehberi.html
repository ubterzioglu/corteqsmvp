🚀 Giriş

CorteQS Fin'de kullanıcı yetkileri kategori → feature → rol hiyerarşisi ile yönetilir. Ayrıca, sahipsiz katalog itemlerinin sahiplik taleplerini kontrol eden bir claim sistemi vardır. Bu rehber her şeyi adım adım açıklıyor.

1️⃣ Üç Katman Modeli

Yetkilendirme sistemi üç katmandan oluşur:

Katman Nerede Tutulur Ne İşe Yarar
Kategori src/lib/features.ts Feature'ları mantıksal gruplara ayırır (Profil, Directory, İletişim, vb.)
Feature feature_catalog tablosu Somut bir yetki (örn. "directory.visible")
Rol roles tablosu Kullanıcı tipi; feature'ları toplu açar/kapar
Override user_feature_overrides tablosu Tek kullanıcıya rol dışı istisna yetki

📋 Kategoriler — Organize Yapı

Kategori, feature'ları admin arayüzünde organize göstermek için kullanılan mantıksal gruptur. 7 kategori vardır:

Simge Kategori Features Amaç
📋 Profil Yönetimi profile.* Profil görüntüleme ve düzenleme
🗂️ Directory directory.* Public directory görünürlüğü
💬 İletişim contact.* İletişim talepleri alma
✍️ İçerik content.* İçerik oluşturma ve düzenleme
🎯 Üretim events.*, offers.*, referral.* Etkinlik, teklif, referral oluşturma
🌐 Platform cadde.*, city.* Platform özel alanları
⚙️ Sistem admin.* Admin kontrolleri

⚠️ Önemli: Kategoriler sadece UI'da organizasyondur. Veritabanında feature'ları yönetmez.

✨ Features — Somut Yetkiler

Feature, bir somut yetki veya işlevsellik. Veritabanında feature_catalog tablosunda tutulur.

Örnekler:

  • profile.view_own → Kullanıcı kendi profilini görebilir
  • directory.visible → Kullanıcı public directory'de listelenebilir
  • events.create → Kullanıcı etkinlik oluşturabilir

Feature'ın Özellikleri

Özellik Açıklama Örnek
Key Unique identifier profile.edit_own
Label Türkçe görünen ad "Kendi Profilini Düzenle"
Global State Sistem genelinde açık mı? true
Scope Role Hangi role(ler) için? * (tüm) veya specific

⚡ Öncelik Sırası — Feature Karar Ağacı

Bir kullanıcı feature X'i kullanabilir mi? Bu karar şu sırayla belirlenir:

1. GLOBAL DURUM
   └─ is_active_globally = FALSE? → ❌ KİMSE KULLANAMIYOR

2. KULLANICI OVERRIDE
   └─ user_feature_overrides = TRUE? → ✅ VERİLİ

3. ROL FEATURE BAYRĞI
   └─ role_feature_flags = TRUE? → ✅ VERİLİ

4. ROL FEATURE VARSAYILANI
   └─ role_feature_defaults = TRUE? → ✅ VERİLİ

SONUÇ: Yukarıdaki ilk kural uygulanır (en spesifik kazanır)

Örnek Senaryolar

Durum Sonuç
Global kapalı, rol açık, override yok ❌ Kapalı
Global açık, rol kapalı, override = açık ✅ Açık
Global açık, rol açık, override = kapalı ⚠️ Kapalı (override kazandı)

Kural: Global toggle kapalıysa, override bile o feature'ı açamaz!

🏷️ Claim Sistemi — Sahiplik Talepleri

Claim sistemi, birinin sahip olduğu bir şeyi (işletme, danışman profili, vb.) sistemde talep etmesine izin verir.

Basit örnek:
ABC Restoran eski bir katalog item'i. Hiç kimse oluşturmadı. Gerçek restaurant sahibi bulundu ve "bu benim işletmem" dedi. Claim sistemi bunu kontrol edip, sahibiliği verir.

Ne Zaman Claim Yapılır?

Durum Neden? Örnek
Eski/İthal Item Sistemde var ama kimse oluşturmadı Eski restoran listesi, ithal edilen işletmeler
Sahipsiz Profil Profil var ama sahibi giriş yapmadı Gerçek danışman, profili buldu
Doğrulama Sahipliği kanıtlamak gerekli Sertifika, ruhsat, belgeler

👣 Claim Adım Adım

1️⃣ Sahip Katalogu Bulur
"ABC Restoran" var mı diye arar.
2️⃣ "Bu Benim" Butonuna Tıklar
"Claim This Business" veya "Bu Benim" butonuna tıklar.
3️⃣ Talep Formu Doldurur
  • Claim Tipi: "ownership" (sahiplik)
  • Kanıt (Evidence): Fotoğraf, sertifika, ruhsat dosyası
  • Not: Neden bu benim? Açıklama
4️⃣ Gönderir
Talep sisteme kaydedilir. Status = pending
5️⃣ Admin İnceliyor ve Onaylarsa
  • ✅ Sahiplik talep edene verilir
  • ✅ Item'in sahibi değişir
  • ✅ Kişi artık o item'i yönetebilir

📖 Claim Senaryoları

Başarılı Senaryo: Restaurant Sahibi

Mehmet'in "Mehmet's Kebab" adında bir restoran var. Sitede bu restaurant zaten var. Mehmet "bu benim" demek istiyor.

1. Mehmet'in Yaptığı:
  • Restoran katalogunu bulur
  • "Bu İşletmem" butonuna tıklar
  • Form doldurur: Ruhsat fotoğrafı, işletme belgesi, açıklama
  • Gönderir
2. Sistem'in Yaptığı:
  • Talep kaydediliyor: catalog_claim_requests tablosuna
  • Status: pending
  • Admin paneline düşüyor
3. Admin Onaylarsa:
  • ✅ Talep status = approved
  • ✅ Restaurant'ın sahibi = Mehmet
  • ✅ Mehmet artık restoran'ı düzenleyebiliyor

Başarısız Senaryo: Kanıtsız Claim

Ahmet bir restoran buldu ve "bu benim" dedi ama kanıt yok.

❌ Admin Reddederse:
  • Kanıt yok → Şüpheli
  • Talep status = rejected
  • Ahmet tekrar deneyebilir (kanıtla)

🔧 Admin: Claim Kontrolü

Durum Anlamı Sonrası
pending Talep yapıldı, admin incelemedi Admin kontrol edebilir
approved Admin onayladı Sahiplik verildi, işlem bitti
rejected Admin reddetti Yeni talep açılabilir
cancelled Talep iptal edildi Yeni talep açılabilir

Admin Neler Yapabilir?

İşlem Ne Yapar? Nerede?
Talep İncele Kanıt ve detayları görebilir /admin/approvals
Approve Sahiplik verilir /admin/approvals → Approve
Reject Talep reddedilir /admin/approvals → Reject
Not Ekle Neden karar verdi not düşebilir /admin/approvals → Review Note

🖥️ Admin Arayüzleri — Kontrol Panelleri

1. /admin/new-member/roles-features — Çapraz Tablo

Ne gösterilir?

  • Satırlar: Tüm features (kategori başlıklarıyla gruplu)
  • Sütunlar: Tüm roller

Ne yapılır?

  • Global toggle: Feature'ı sistem genelinde aç/kapat
  • Hücre toggle: İlgili rolde o feature açık mı?

2. /admin/new-member/users-roles — Kullanıcı Roller

Bireysel kullanıcılara rol atama, effective features özeti gösterme

3. /admin/new-member/overrides — Kullanıcı İstisnaları

Tek kullanıcıya, rolünden bağımsız olarak feature açıp kapatma

💡 İpucu: Override verirken neden alanını doldurun. Audit logda kritiktir!

📖 Pratik Admin Senaryoları

Senaryo 1: Yeni İçerik Üreticisi Ekle

Amaç: BloggerX'i blogger-vlogger-youtuber rolüyle ekle, içerik oluşturabilsin.

  1. /admin/new-member/users-roles → BloggerX → Rol = blogger-vlogger-youtuber
  2. /admin/new-member/roles-features → content.create satırı → sütun = ✅
  3. BloggerX oturum açınca content.create = true döner
  4. ✅ BloggerX içerik oluşturabilir

Senaryo 2: Danışman'ı Directory'de Gizle

Amaç: DoctorX (danisman) directory'de görünmesin.

  1. /admin/new-member/overrides → DoctorX seç
  2. directory.visible → ❌ disable
  3. Neden: "Mahremiyet isteği"
  4. ✅ DoctorX directory'de gizli kalır

Senaryo 3: Tüm Kullanıcılara Etkinlik Oluşturma Kapalı

Amaç: events.create feature'ını system genelinde kapat (geçici).

  1. /admin/new-member/roles-features → events.create satırı
  2. Global toggle → ❌ kapalı
  3. ✅ Tüm roller için etkinlik oluşturma kapalı

🔧 Teknik: Tablo İlişkileri

feature_catalog (key PK)
  ├─ key (string) - Unique identifier
  ├─ label - Türkçe adı
  ├─ is_active_globally ← GLOBAL KILIT
  └─ scope_role - '*' veya specific role_key

role_feature_flags (role_id, feature_key) PK
  ├─ role_id FK
  ├─ feature_key FK
  └─ is_enabled ← ROL BAYRĞI

user_feature_overrides (user_id, feature_key) PK
  ├─ user_id FK
  ├─ feature_key FK
  ├─ is_enabled ← KULLANICI ÖZ YETKI
  └─ reason - Audit için

catalog_claim_requests (id PK)
  ├─ item_id FK
  ├─ requested_by_user_id FK
  ├─ claim_type (ownership)
  ├─ evidence (JSON)
  ├─ status (pending|approved|rejected|cancelled)
  └─ reviewed_by_user_id FK (admin tarafından)

📡 RPC Fonksiyonları

get_current_user_features()

Şu anki kullanıcı için feature'ları çözen RPC:

SELECT
  feature_key,
  is_enabled = (
    is_active_globally AND COALESCE(
      user_override.is_enabled,
      role_flag.is_enabled,
      role_default.is_enabled,
      false
    )
  ) AS resolved_enabled
FROM feature_catalog
WHERE scope_role = user_role OR scope_role = '*'

submit_catalog_claim_request()

Kullanıcı tarafından talep oluşturma:

  • target_item_id: Hangi item?
  • claim_type: 'ownership'
  • evidence: JSON kanıtlar
  • note: Açıklama

review_catalog_claim_request()

Admin tarafından talep inceleme ve karar:

  • target_claim_request_id: Hangi talep?
  • decision: 'approved' | 'rejected' | 'cancelled'
  • review_note: Açıklama

✅ Özet

CorteQS Yetkilendirme Sistemi üç başlıkta çalışır:

  1. Kategori-Rol-Feature: Admin kategorileri görerek, rol başına feature'ları kontrol eder
  2. Karar Ağacı: Global → Override → Rol → Fallback sırası ile feature kullanılabilirliği karar verilir
  3. Claim Sistemi: Kullanıcı sahipsiz item'lere talep yapabilir, admin inceler ve onaylar

🎯 Amaç: Flexible yet controlled authorization with ownership verification for catalog items.
