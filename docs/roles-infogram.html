01

Büyük Resim — Neden İki Sistem Var?

Her sistemi anlamadan kod yazmak tehlikeli olabilir.

⚠️
Junior'a kritik uyarı
Bu projede iki ayrı auth/rol sistemi aynı anda çalışıyor. Birbirini bilmiyor gibi davranıyor ama aslında paralel. Birini atlarsanız bug çıkar. Her feature'da hangisini kullandığınızı bilin.

Legacy Sistem (Eski — Hâlâ Aktif)

Basit ama sınırlı

Kullanıcının bir profile_type alanı var. 6 sabit değerden biri. Admin olmak için ayrı bir admin_users tablosuna eklenmek gerekiyor.

6 sabit rol genişletilemiyor hâlâ prod'da aktif

RolesGo Sistemi (Yeni — MVP Mayıs 2026)

Esnek ve genişletilebilir

82 rol, 36 feature, attribute yönetimi, onay iş akışı, audit log. Migration yazmadan yeni rol eklenebiliyor. Tam bir RBAC + feature flag sistemi.

82 rol migration'sız genişler audit trail var
💡
Nasıl birlikte çalışıyorlar?
Legacy sistem, eski sayfalar tarafından hâlâ kullanılıyor. is_admin() SQL fonksiyonu her iki sistemi de kontrol ediyor (admin_users tablosu VE profiles.platform_role = 'admin'). Yeni feature geliştirirken RolesGo sistemini kullanın.

03

RolesGo Sistemi — Yeni Yapı

Mayıs 2026'da eklendi. Yeni feature'lar için bu sistemi kullanın.

✅
Neden daha iyi?
Yeni bir rol tipi eklemek için migration yazmak gerekmiyor. Sadece roles tablosuna bir satır eklemek yeterli. Feature'ları da aynı şekilde feature_catalog'a ekleyebilirsiniz.

Roller Tablosu

id UUID -- PRIMARY KEY
key TEXT UNIQUE -- rol tanımlayıcısı
label TEXT -- görüntü adı
descriptionTEXT
sort_order INT -- aile gruplarını ayırır
is_active BOOLEAN
created_at TIMESTAMPTZ

Tablo adı: public.roles — 82 satır

Kullanıcı ↔ Rol Atama Tablosu

user_id UUID -- PK + FK → auth.users
role_id UUID -- FK → public.roles
updated_by UUID -- kim atadı?
created_at TIMESTAMPTZ
updated_at TIMESTAMPTZ
-- 1 kullanıcı = 1 rol (one-to-one)
-- user_id PRIMARY KEY olduğu için

Tablo adı: public.user_role_assignments

ℹ️
sort_order neden önemli?
sort_order 10–60 arası = legacy roller. 1000+ = yeni RolesGo rolleri. Aynı tabloda ama farklı "aileler". Yeni rol eklerken doğru aile range'ini kullanın.

Feature Catalog — 36 Feature

"Bu kullanıcı bu özelliği kullanabilir mi?" sorusunun cevabı burada.

💡
Feature nedir?
Bir kullanıcının yapabildiği veya görebildiği şeylerin anahtarları. Örneğin cadde.access = Cadde sayfasına girebilir mi? events.create = Etkinlik oluşturabilir mi? Her feature için "açık/kapalı" kararı verilir.

Feature Çözümleme Hiyerarşisi

"Bu feature aktif mi?" sorusu 5 adımda cevap bulur. Sıra önemli!

⚠️
Junior için kritik
Bir feature'ın açık olup olmadığını siz hesaplamayın. useFeatureFlags() hook'unu veya get_current_user_features() RPC'sini kullanın. Bu 5 adımlı mantık zaten orada uygulandı.
1
Kişisel Override (En Yüksek Öncelik)
Admin, bu kullanıcı için özellikle bir istisna tanımladıysa — bu kazanır. Her şeyi ezer.
user_feature_overrides
↓ bulunamazsa
2
Rol Bazlı Ayar
Kullanıcının rolü için bu feature açık/kapalı olarak ayarlandıysa — bu kullanılır.
role_feature_flags
↓ bulunamazsa
3
Legacy Profile Type Varsayılanı
Eski sistem: kullanıcının profile_type'ına göre default değer. Legacy kaldıkça burada kalacak.
role_feature_defaults
↓ bulunamazsa
4
Global Kapatma Kontrolü
Feature tüm sistemde kapalıysa (is_active_globally = false) — herkes için kapalı.
feature_catalog
↓ hâlâ bulunamazsa
5
Varsayılan: Kapalı
Hiçbir kayıt bulunamadıysa → feature kapalı sayılır.
false (fallback)

Attribute Sistemi

Kullanıcı profilindeki "ek bilgiler" — esnek, rol bazlı, onay destekli.

💡
Attribute nedir?
Profil tablosuna kolon eklemek yerine, dinamik "özellikler" sistemi. Örneğin "LinkedIn URL", "Uzmanlık Alanı", "CV Dosyası". Her attribute, hangi rol için gerekli/isteğe bağlı/görünür olduğunu ayrıca tanımlar.

08

Onay (Approval) State Machine

Hassas işlemler doğrudan gerçekleşmiyor — önce admin onayı gerekiyor.

🔄
State Machine nedir?
Bir istek sadece belirli durumlar arasında geçiş yapabilir. PENDING → APPROVED veya PENDING → REJECTED olabilir ama APPROVED → PENDING olamaz. Bu kurallar database'de zorlanıyor.

Frontend Entegrasyonu

Kod yazarken hangi hook ve component'i kullanacaksınız?

useFeatureFlags()

src/hooks/useFeatureFlags.ts — Supabase RPC'yi çağırır ve feature durumlarını döndürür.

// Kullanım
const {
isFeatureEnabled,
getFeatureSource,
featureMap,
isLoading,
refreshFeatures
} = useFeatureFlags()
// Kontrol
isFeatureEnabled('cadde.access')
// → true | false

<RequireFeature />

src/components/auth/RequireFeature.tsx — Feature gate component. Feature kapalıysa fallback gösterir.

{/* Kullanım */}
<RequireFeature
feature="cadde.access"
fallback={<UpgradePrompt/>}
>
<CaddeComponent/>
</RequireFeature>

Önemli Dosya Referansları

Dosya Ne İçeriyor Ne Zaman Bakılır

Legacy vs RolesGo — Tam Karşılaştırma

Hangi sistemi ne zaman kullanmalıyım?

Özellik Legacy Sistem RolesGo Sistemi
Rol sayısı 6 (sabit enum değeri) 82+ (tabloda satır)
Yeni rol eklemek ✗ Migration gerekir ✓ Sadece tablo INSERT
Rol atama user_profiles.profile_type user_role_assignments tablosu
Feature kontrolü role_feature_defaults role_feature_flags per role_id
Kişisel override ✗ Yok ✓ user_feature_overrides + reason
Profil attribute Tablo kolonları (sabit) Dinamik attribute_catalog
Görünürlük ✗ Desteklenmiyor ✓ public / private / admin_only
Onay iş akışı ✗ Manuel admin işlemi ✓ approval_requests state machine
Admin tespiti Sadece admin_users tablosu admin_users + profiles.platform_role
Audit trail ✗ Yok ✓ admin_audit_logs (before/after)
Genişletilebilirlik ✗ Schema değişikliği gerekir ✓ Migration'sız eklenebilir
Mevcut durum Aktif — korunuyor Aktif — büyüyor
🚨
Ne zaman hangi sistemi kullanmalısınız?
Mevcut sayfalara dokunuyorsanız: Hangi sistemi kullandığını önce kontrol edin. profile_type görüyorsanız → legacy. useFeatureFlags/RequireFeature görüyorsanız → RolesGo.

Yeni feature yazıyorsanız: Her zaman RolesGo. feature_catalog'a key ekleyin, RequireFeature kullanın.

Admin kontrolü yapıyorsanız: is_admin() SQL fonksiyonunu kullanın — her iki sistemi de kontrol ediyor.
