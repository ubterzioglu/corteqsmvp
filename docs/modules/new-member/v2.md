# CorteQS Roles Matrix — AI Technical Continuation Brief

## 0. Bu Dokümanın Kullanım Amacı

Bu doküman, `roles(1).xlsx` dosyasındaki role / feature / UI matrix yapısını AI agent, developer veya teknik ekip tarafından devam geliştirilebilir hale getirmek için hazırlanmıştır.

Excel dosyası doğrudan kod üretmek için yeterli değildir; çünkü aynı tabloda şu farklı kavramlar birlikte durmaktadır:

1. **UI görünüm parçaları**  
   Örnek: ön kart rozeti, detay kart sekmesi, profil görseli.

2. **Gerçek permission / capability**  
   Örnek: mesaj gönderebilme, etkinlik yönetebilme, iş ilanı oluşturabilme.

3. **Profil attribute alanları**  
   Örnek: bio, tagline, diller, yaşadığı ülkeler, business WhatsApp no.

4. **Alt kategori / alt tip taxonomy değerleri**  
   Örnek: Consultant alt kategorileri, Business alt tipleri.

5. **Admin ve moderation modülleri**  
   Örnek: onaylar, kullanıcılar, elçi başvuruları, şikayetler.

Bu nedenle devam geliştirme yapılırken Excel satırları tek bir `feature_flag` tablosuna körlemesine basılmamalıdır. Önce satırlar doğru teknik sınıfa ayrılmalı, sonra uygun tabloya yazılmalıdır.

## 3. Kritik Normalizasyon Kuralı

```txt

```

Karar tablosu:

| Consultant alt uzmanlık ise | `taxonomy_options` |
| Business classic/online/startup ise | `taxonomy_options` |

## 5. Feature Key Standardı

Önerilen key formatı:

```txt
scope.semantic_name
```

Excel bölümüne göre önerilen scope:

| Excel Bölümü | Scope Prefix | Örnek |
|---|---|---|
| Ön Kart | `preview.*` | `preview.profile_logo_image` |
| Detay Kart | `detail.*` | `detail.events_tab` |
| Kontrol Paneli | `dashboard.*` | `dashboard.messages_tab` |
| Alt-Kategori | `consultant_category.*` | `consultant_category.legal` |
| Alt-Tip | `business_subtype.*` | `business_subtype.online` |

Not:

- `preview.*` ve `detail.*` çoğunlukla permission değil, UI section/component kuralıdır.
- `dashboard.*` satırlarının bir kısmı gerçek permission’dır.
- `consultant_category.*` ve `business_subtype.*` feature değil, taxonomy option olmalıdır.

## 6. Teknik Tablo Tasarımı

## 6.1 `feature_catalog`

Gerçek aksiyon / yetki / modül erişimi için kullanılmalıdır.

Önerilen kolonlar:

```sql
id uuid primary key default gen_random_uuid(),
key text not null unique,
label text not null,
description text,
scope text not null, -- profile, dashboard, messaging, events, offers, admin, referral
feature_type text not null, -- capability, module, tab, action, admin
default_enabled boolean not null default false,
requires_approval boolean not null default false,
is_active boolean not null default true,
metadata jsonb not null default '{}'::jsonb,
sort_order integer not null default 100,
created_at timestamptz not null default now(),
updated_at timestamptz not null default now()
```

Kullanılacağı örnekler:

- `dashboard.tab_mesaj_kutusu`
- `dashboard.tab_etkinlikler`
- `dashboard.tab_ilanlar_is_ilanlari`
- `dashboard.tab_onaylar`
- `dashboard.tab_kullanicilar`
- `dashboard.tab_elci_yonetimi`

---

## 6.2 `profile_section_catalog`

Public ön kart, detay kart ve profil görünüm parçaları için kullanılmalıdır.

Önerilen kolonlar:

```sql
id uuid primary key default gen_random_uuid(),
key text not null unique,
section_area text not null, -- preview_card, detail_card
label text not null,
description text,
component_name text,
data_source text, -- user_profiles, user_profile_attributes, derived, external
is_active boolean not null default true,
metadata jsonb not null default '{}'::jsonb,
sort_order integer not null default 100,
created_at timestamptz not null default now(),
updated_at timestamptz not null default now()
```

Kullanılacağı örnekler:

- `preview.konum`
- `preview.profil_logo_gorseli`
- `preview.google_rating_badge`
- `detail.sosyal_medya_plus_linkedin`
- `detail.tab_etkinlikler`
- `detail.hakkinda_bio`

---

## 6.3 `role_profile_section_rules`

Hangi profil bölümünün hangi rolde görüneceğini belirler.

```sql
id uuid primary key default gen_random_uuid(),
role_id uuid not null references roles(id) on delete cascade,
section_id uuid not null references profile_section_catalog(id) on delete cascade,
is_enabled boolean not null default true,
requires_approval boolean not null default false,
visibility_default text not null default 'public',
sort_order integer not null default 100,
metadata jsonb not null default '{}'::jsonb,
created_at timestamptz not null default now(),
updated_at timestamptz not null default now(),
unique(role_id, section_id)
```

## 7. UI Rendering Contract

## 7.1 Ön Kart Renderer

Frontend, ön kartı role göre hardcode etmemelidir.

Önerilen akış:

```txt
get_public_profile_card(profile_id)
  → profile basic info
  → role key
  → enabled preview sections
  → public-visible attributes
  → derived badges
  → render ordered components
```

Ön kartta data kaynağı ayrımı:

| UI Parçası | Data Source |
|---|---|
| Konum | `user_profiles.country`, `user_profiles.city` veya attribute |
| Profil/Logo | `profile_photo_url` |
| Follower / üye sayısı | derived counter |
| Google Rating | external/manual rating aggregate |
| Kategori etiketi | taxonomy / role attribute |
| Rozetler | derived flags |
| Mesaj ikonu | feature + contact settings |
| Mini takvim | event participation / event ownership |

---

## 7.2 Detay Kart Renderer

Detay kart sekmeleri de role göre hardcode edilmemelidir.

Önerilen akış:

```txt
get_public_profile_detail(profile_id)
  → enabled detail sections
  → role-specific attributes
  → contact visibility
  → events / jobs / offers / coupons relations
  → render tabs and sections
```

Önemli kurallar:

- Consultant için WhatsApp no doğrudan genel iletişimden çıkarılmalı; business WhatsApp ayrı alan olmalı.
- Embedded iframe harita kullanılmamalı; harita linki / directions yaklaşımı tercih edilmeli.
- AI görüşme ve canlı görüşme CTA’ları MVP’de disabled / yakında badge ile gösterilmeli.
- Gayrimenkul ilanları yalnızca Consultant alt kategorisi `gayrimenkul` ise görünmeli.

---

## 7.3 Kontrol Paneli Renderer

Kontrol paneli role + feature + override ile render edilmelidir.

Önerilen akış:

```txt
get_current_user_features()
  → role flags
  → user overrides
  → approval status
  → dashboard navigation items
```

Dashboard nav item gösterim kuralı:

```txt
show item if:
  feature is active
  AND effective user permission is true
  AND required approval status is approved or not required
```

Admin için:

```txt
is_admin(auth.uid()) = true
```

kontrolü RLS/RPC seviyesinde yapılmalıdır.

## 8. Excel Matrix Değerlendirme Notları

## 8.1 Güçlü Taraflar

- Role bazlı UI kapsamı oldukça netleşmiş.
- Public card, detail card ve dashboard ayrımı doğru yönde.
- Admin panel ihtiyaçları görünür hale gelmiş.
- Consultant alt kategorileri ve Business alt tipleri ayrılmış.
- “Yakında”, “V2”, “Faz-2” gibi ürün fazları notlara yazılmış.
- Variant notları ileride UI component tasarımına yardımcı olacak.

## 8.2 Teknik Riskler

| Risk | Açıklama | Çözüm |
|---|---|---|
| Feature / UI / Attribute karışması | Matrix tek tablo gibi görünse de satırlar farklı teknik sınıflara ait | Import öncesi sınıflandırma yapılmalı |
| Hardcoded role UI | Her rol için ayrı component yazılırsa sistem büyüyemez | Catalog-driven renderer |
| Hassas veri sızıntısı | WhatsApp, telefon, email public’e düşebilir | visibility + approval + RLS |
| Admin bypass | Frontend guard tek başına yeterli olmaz | RPC + RLS + audit log |
| Faz-2 özelliklerin erken açılması | AI görüşme / canlı görüşme aktif sanılabilir | feature status: disabled/coming_soon |
| Taxonomy’nin feature sanılması | Consultant alt kategorileri permission değildir | taxonomy table |
| Matrix satır adlarının key yapılması | Türkçe label değişirse sistem kırılır | stable English/system key |

## 10. RPC Kontratı

Önerilen RPC fonksiyonları:

```txt
get_current_user_features()
get_current_user_dashboard()
get_public_profile_card(profile_id)
get_public_profile_detail(profile_id)
submit_approval_request(request_type, payload)
admin_review_approval_request(request_id, decision, admin_note)
admin_set_role_feature_flag(role_key, feature_key, is_enabled)
admin_set_user_feature_override(user_id, feature_key, is_enabled, reason)

admin_set_profile_section_rule(role_key, section_key, is_enabled)
admin_set_attribute_rule(role_key, attribute_key, rule_payload)
```

Önemli not:

`get_public_profile_card` ve `get_public_profile_detail` gibi read RPC’leri hassas veri filtrelemesini backend tarafında yapmalıdır. Frontend’e önce tüm data gönderilip sonra saklanmamalıdır.

## 11. RLS Kuralları

Genel kurallar:

```txt
users can read own profile
users can update own editable attributes
users cannot update own role directly
users cannot update own feature flags directly
users cannot approve own approval requests
public can read only approved public profile projection
admins can read/write through RPC
admin direct table write should be minimized
```

Kritik hassas veri alanları:

```txt
email
phone
whatsapp
business_whatsapp
license_upload_url
approval payload
admin notes
```

Bu alanlar public query projection içinde asla default olarak yer almamalıdır.

## 20. Kısa Yönetici Özeti

Bu matrix, CorteQS için çok iyi bir ürün haritası çıkarıyor; ancak teknik olarak tek bir feature flag listesi değildir.

Doğru yaklaşım:

```txt
Public görünüm parçaları → profile_section_catalog
Gerçek yetkiler → feature_catalog
Profil veri alanları → attribute_catalog
Alt kategori/tipler → taxonomy_options
Onay gerektirenler → approval_requests
Admin işlemleri → RPC + audit log
```

Bu ayrım yapılırsa sistem hem MVP’de hızlı ilerler hem de ileride yeni role, yeni attribute, yeni feature ve yeni dashboard tab eklemek kontrollü hale gelir.
