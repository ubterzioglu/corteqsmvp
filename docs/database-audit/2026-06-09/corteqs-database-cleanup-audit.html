14. Migration Gecmisi Analizi

192 onceki migration + 4 yeni = 196. Tumu supabase/migrations/ altinda append-only sirada, schema_migrations ile senkron.

Bu denetimde eklenen 4 migration governance standardina uydu: tek-amacli, additive-first (P0 REVOKE + RLS, indeksler CONCURRENTLY), her biri bagimsiz geri alinabilir.

Migration'lar uretimde sirayla replay edilir — asla silinemez, duzenlenemez, yeniden siralanamaz. Yalnizca yeni migration eklenir.

15. Naming Convention Rehberi

Tam rehber: database-migration-roadmap.md §1. Ozet (yalnizca YENI nesneler icin — mevcut nesneleri toplu yeniden adlandirma):

  • Tablo: snake_case, modul prefix, koleksiyonlar icin cogul. Turkce domain kokleri korunur (muhasebe, cadde, lansman, gorevler).
  • PK: id uuid DEFAULT gen_random_uuid().
  • FK kolonu: <referenced_singular>_id; FK her zaman indekslenir.
  • RPC: verb-first; SECURITY DEFINER SET search_path = public, pg_temp pinlemeli.
  • Admin RPC: admin_ prefix; EXECUTE yalnizca authenticated (anon degil); iceride is_admin() gate.
  • View: v_ prefix; her zaman security_invoker = on.
  • Index: idx_/uq_; prod'da CONCURRENTLY.
  • Migration dosya adi: YYYYMMDDHHMMSS_<module>_<purpose>.sql.
Hard rule: Mevcut 154 tablo / 157 RPC / 108 FK frontend + RLS tarafindan referanslaniyor. Rename = breaking change. Gerekirse compatibility window (eski ad + yeni alias view) kullan, asla tek adimda rename+drop yapma.

16. Migration Governance Standardi

Tam standart: database-migration-roadmap.md §2. Cekirdek ilkeler:

  1. Kucuk, tek-amacli migration.
  2. Additive-first — eklemeden once kaldirma yapma; kaldirma son care.
  3. Her rename oncesi compatibility window.
  4. Her drop oncesi kullanim kaniti (0 satir beklenen yerde, 0 kod referansi, 0 pg_stat_statements isabeti, yedek).
  5. Her constraint oncesi veri-kalite SELECT'i.
  6. Her index oncesi duplicate/kapsama kontrolu + CONCURRENTLY.
  7. Her RLS degisikliginde staging'de policy-matrix testi.
  8. Her RPC degisikliginde EXECUTE-privilege kontrolu.
  9. Her destructive degisiklik icin rollback yolu zorunlu.
  10. Big-bang temizlik yok — fazla; her faz bagimsiz geri alinabilir.
  11. idx_scan=0 tek basina index drop nedeni degildir (restart'ta stat sifirlanir).

sql-drafts/ dosyalari asla supabase/migrations/'a kopyalanmaz — onaylanan taslak icin DBA taze, gozden gecirilmis bir migration yazar (up/down ile).

20. Metodoloji ve Sinirlamalar

  • Veri kaynagi: Canli uretim (injprdrsklkxgnaiixzh) read-only metadata sorgulari (pg_catalog, pg_policies, pg_stat_user_indexes, information_schema) IPv4 pooler uzerinden.
  • Uygulanan degisiklik: Yalnizca 4 migration (P0 guvenlik + grant sertlestirme + view normalize + FK indeksleri) — hepsi additive/REVOKE, geri alinabilir.
  • .from() siniflandirma siniri: Frontend referans sayimi yalnizca dogrudan supabase.from() kullanimini yakalar; RPC/trigger erisimi ayri sayilir. "0 from" ≠ "kullanilmiyor".
  • Docker yok: CLI db dump yerine dogrudan pooler metadata sorgulari kullanildi.
  • Satir sayilari: Est_Rows alani tahminidir (-1 = sayilmadi/belirsiz); kritik drop kararlari icin canli COUNT dogrulamasi gerekir.
  • Skorlar muhafazakar: 0-100 saglik skorlari kanita dayali ve ±10 yargi payi icerir.
